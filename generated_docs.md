### 1. 변경 요약

주요 변경 사항은 벤치마크 시스템을 대폭 개선하여 새로운 의미론적 리콜(Semantic Recall) 평가 방식을 도입하고, 이에 필요한 `numpy` 및 관련 로직을 추가하며, 테스트 데이터셋 파일 관리를 변경하고 불필요한 `generated_docs.md` 파일을 제거하는 것입니다.

### 2. 상세 내용

*   **벤치마크 시스템 대대적인 개선 (`tests/benchmark.py`):**
    *   **의미론적 리콜 도입**: 모델의 출력과 Ground Truth 간의 의미적 유사도를 평가하는 새로운 `calculate_semantic_recall` 함수가 추가되었습니다. 이 함수는 단어를 토큰화하고, 불용어(stop words)를 제거한 뒤, `text-embedding-004` 모델을 활용하여 단어 임베딩(word embeddings)을 통해 의미적 일치도를 측정합니다.
    *   **임베딩 및 유사도 계산 기능 추가**: 특정 단어에 대한 임베딩을 가져오는 `get_word_embeddings` 함수와 두 벡터 간의 최대 코사인 유사도를 계산하는 `compute_max_similarity` 함수가 구현되었습니다.
    *   **데이터셋 관리 변경**: 기존의 단일 `dataset.py`에서 `dataset_easy.py` 및 `dataset_normal.py`와 같이 여러 데이터셋 파일을 불러오도록 변경되었고, `run_benchmark` 함수가 `test_dataset`을 인자로 받도록 유연성이 향상되었습니다.
    *   **로깅 및 결과 형식 개선**: 벤치마크 결과에 Semantic Recall 점수, 소요 시간, 누락된 토큰 목록 등을 포함한 상세 정보가 추가되어 가독성이 높아졌습니다.
    *   **환경 변수 로드**: `dotenv`를 사용하여 환경 변수를 로드하도록 추가되었습니다.
*   **의존성 및 설정 파일 업데이트:**
    *   `requirements.txt`에 과학 계산을 위한 `numpy` 라이브러리가 새로 추가되었습니다.
    *   `.gitignore`에서 `output.txt` 및 `tests/dataset.py` 규칙이 제거되었고, `tests/dataset_*.py` 규칙이 추가되어 테스트 데이터셋 파일의 패턴 관리가 변경되었습니다.
    *   `.github/workflows/auto-docs.yml` 파일 내 Python 캐시 관련 주석이 "의존성 캐싱"으로 명확하게 업데이트되었습니다.
*   **불필요 파일 삭제:**
    *   `generated_docs.md` 파일이 프로젝트에서 완전히 삭제되었습니다. (이 파일은 이전 변경 사항을 설명하는 자동 생성 문서였습니다.)

### 3. 기술적 영향

*   **평가 정확도 및 깊이 향상**: AI 모델의 문서 생성 품질을 단순히 토큰 수나 문자열 일치도뿐만 아니라 의미론적으로 평가할 수 있게 되어, 실제 사용자 관점에서의 모델 성능에 대한 더 정확하고 깊이 있는 통찰을 얻을 수 있습니다. 이는 모델 개선 방향을 설정하는 데 매우 큰 도움이 될 것입니다.
*   **외부 의존성 및 복잡성 증가**: `numpy`와 같은 추가 라이브러리 및 임베딩 API 호출 로직 도입으로 프로젝트의 외부 의존성과 코드 복잡성이 증가했으며, 이는 환경 설정 및 배포 시 추가적인 고려사항이 될 수 있습니다.
*   **모듈화 및 유연성 증대**: `dataset` 파일의 분리 및 벤치마크 함수의 유연성 증대는 다양한 테스트 케이스와 데이터셋을 쉽게 적용하고 관리할 수 있게 하여 벤치마크 시스템의 확장성과 재사용성을 높입니다.
*   **리소스 사용량 증가 가능성**: 임베딩을 이용한 의미론적 평가는 모델 호출을 추가로 발생시킬 수 있어 벤치마크 실행 시간과 API 호출 비용에 영향을 미칠 수 있습니다. 불용어(`STOP_WORDS`) 설정 등은 이러한 오버헤드를 줄이려는 노력으로 보입니다.
*   **코드베이스 정리**: `generated_docs.md` 파일의 삭제는 자동 생성된 문서의 처리 방식이 변경되었거나, 더 이상 해당 파일에 직접 문서를 저장할 필요가 없음을 나타내며, 프로젝트의 전반적인 코드베이스를 간결하게 유지하는 데 기여합니다.
